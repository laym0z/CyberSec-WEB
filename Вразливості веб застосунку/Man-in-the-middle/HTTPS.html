HTTPS

HTTPS - це захищена версія HTTP, яка буде передавати дані між браузером та веб сайтом у зашифрованому вигляді. Він використовує TLS та шифрує з'єднання завдяки асиметричному шифруванню, яке використовує приватний та публічний ключи. Публічний ключ буде надсилатися веб сервером клієнтові, як тільки той спробує під'єднатися до веб сайту.

Публічний ключ (знаходиться на стороні клієна) - шифрує дані перед відправкою на сервер Приватний ключ (знаходиться на сервері) - розшифровує отримані дані від клієнта

Для встановлення HTTPS з'єднання веб сайт повинен мати TLS/SSL сертифікат, який підтверджує, що конкретний провайдер є тим, за кого себе видає. Такі сертифікати видаються різними хостингами за деяку плату, а деякі видають навіть безкоштовно на власних умовах.

Але існують і так звані самопідписані сертифікати, які створюються власниками веб сайту. Сайти, які використовують такі сертифікати, не будуть викликати довіру у браузерів, тому відповідно будуть помічатися як небезпечні. Для демонстрації мій застосунок використовуватиме саме такий сертифікат, його можна згенерувати за допомогою openssl:

openssl req -x509 -newkey rsa:4096 -keyout key.pem -out cert.pem -sha256 -days 365 -nodes

Згенерований вище сертифікат житиме цілий рік. Також можна встановити додатковий захист приватного ключа, встановивши кодову фразу, але у моєму випадку він не захищений (опція -nodes)

Перед тим, як встановити захищений протокол на веб сайт, я продемонструю перехоплення запиту на логін через HTTP протокол. Для TLS/SSL з'єднання характерне рукостискання на початку комунікації, для встановлення захищеного з'єднання, але як видно на зображенні нижче - рукостискання немає (окрім загального для початку комунікації), відповідно, і з'єднання не захищене. Для того, аби переключитися на захищене з'єднання, треба надати моєму застосунку згенеровані мною сертифікати:

Для Gunicorn

--certfile=/var/www/demo-web-app/app/cert/cert.pem \
--keyfile=/var/www/demo-web-app/app/cert/key.pem \

Але мій застосунок буде працювати на локальному порту, в той час як nginx буде слугувати як proxy та пересилати всі запити на цей локальний порт. Тому згенеровані ключ та сертифікат треба перемістити в директорію /etc/nginx/ssl. Доступ до них повинен мати лише root користувач.

В конфігураційному файлі мого застосунку для nginx (/etc/nginx/sites-available/flaskapp.conf) варто додати наступний вміст:

server {
    listen 443 ssl;

    ssl_certificate     /etc/nginx/ssl/cert.pem;
    ssl_certificate_key /etc/nginx/ssl/key.pem;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers on;

    location / {
        proxy_pass http://127.0.0.1:8000;
    }
}

server {
        listen 80 default_server;
        return 301 https://$host$request_uri;
}

За підсумком - це конфіг, в якому вказано порт для прослуховування, в моєму випадку - це 443 порт, який є стандартним для HTTPS протоколу. Далі надається сертифік, який був згенерований мною ж (само підписаний), встановлення протоколу та на кінець - встановлення порту, на який буде пересилатися весь вхідний трафік.

Другий блок відповідає за перенаправлення з 80 порту на 443. Для коректної роботи варто прибрати стандартний конфіг файл від nginx, де вже вказане прослуховування на 80 порт, оскільки у мене немає доменного імені, тому і розпізнавати різні з'єднання не можливо:

sudo unlink /etc/nginx/sites-enabled/default

Після збереження конфігурації можна перевірити її на валідність наступною командою:

sudo nginx -t

Для завершення треба залінкувати створений конфіг у визначену директорію:

sudo ln -s /etc/nginx/sites-available/flaskapp.conf /etc/nginx/sites-enabled/

Після перезапуску все має почати працювати:

sudo systemctl restart nginx

Тепер якщо подивитися на трафік під час надсилання запиту на логін, то можна побачити, що передані дані зашифровані завдяки TLS/SSL з'єднанню і також видно, як сервер та клієнт встановлюють захищене з'єднання: Клієнт надсилає пакет "Client Hello" а сервер відповідає "Server Hello", і як тільки захищене з'єднання встановлене - відбувається передача даних:
